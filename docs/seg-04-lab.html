Lab: Segmentation Configurator

CloudWorks case study — Segmentation configurator exercise

Lab 45 min Hands-On
📋 Answer Key

This lab is an exercise — there are no answers on this page. Review the CloudWorks case study handout, navigate through the configurator pages, and fill in inputs based on the case study data. The answer key is available in the Appendix → and will be reviewed during the facilitator-led debrief.

Exercise Overview

Using the CloudWorks case study handout, complete the Segmentation section of the configurator. Work through the pages below in order, making configuration decisions based on the case study data. After each configurator page, hit Apply / Calculate before moving on.

💡 Tip

After each configurator page, hit Apply / Calculate before moving on. If a step asks you to navigate to a specific page and the page name differs slightly in your instance, use your best judgment — configurator page names may vary slightly between versions.

Part 1: Segmentation Configuration

Navigate to the Segmentation section of the configurator. Complete these exercises in order.

S1
Data Hub Setup — Page 1.01
⏱ 10 min
Objective Review and capture the firmographic account attributes that will drive segmentation. Ensure all quantitative fields have banded counterparts enabled for the rule engine, and that list-formatted attributes are correctly tagged.
  1. 1 Navigate to Page 1.01: Data Hub Setup.
  2. 2 Review the key firmographic fields available for the CloudWorks account data: refer to the case study handout for the full attribute list.
  3. 3 Capture all relevant segmentation attributes/fields — identify which attributes will be used for rule dimensions, scoring, and UCP (Unacceptable Customer Profile) exclusion.
  4. 4 For each numeric/quantitative field you intend to use (e.g., Employee Count, Annual Revenue): ensure an additional attribute exists for "banding" (e.g., Employee Band, Revenue Band). Enable the "Enable to Calculate" toggle on the banded attribute so the list is generated correctly at application generation.
  5. 5 Click Apply / Calculate before moving on.
💡 Tip

Raw numeric values (like Employee Count = 1,547) cannot be used as rule dimensions. Only list-formatted banded attributes (like Employee Band = "1K–5K") work. If a banded attribute does not yet exist in the data, flag it — it must be created in the source data before the configurator session.

📸 Screenshot needed: Page 1.01 — Data Hub Setup with data objects table populated (not blank/loading). Should show firmographic attributes with banded fields and "Enable to Calculate" toggles visible.
S2
Segmentation Configuration — Rule-Based & Score-Based
⏱ 15 min
Objective Configure the rule-based and score-based segmentation parameters, selecting account attributes, scoring dimensions, and reporting setup based on the CloudWorks case study.
📝 Note: Segment Names and Count

The number of segments and their names (e.g., Strategic, Named, Commercial, SMB) are not configured here. They are defined by admins after application generation via placeholder slots. Do not look for a segment naming input on this page.

  1. 1 Navigate to the correct Segmentation Configuration page.
    [SME CONFIRMATION NEEDED: Exact configurator page name/path for Segmentation rule-based and score-based setup]
  2. 2 Select account attributes for Rule-Based Segmentation — the attribute list available for selection is populated from your S1 Data Hub Setup step. Choose the attributes relevant to CloudWorks's segmentation rules based on the case study.
  3. 3 Select Account Score to Segment mapping dimensional drivers — at what level may the score-to-segment threshold differ? (e.g., thresholds may differ by region or industry). Select the applicable dimensions from the case study.
  4. 4 Select Categorical Metrics to score account — choose the qualitative account attributes (list-formatted) that will be used as scoring inputs.
  5. 5 Select Numerical Metrics to score account — choose the quantitative attributes (banded) that will be used as scoring inputs. These should correspond to the banded attributes you enabled in S1.
  6. 6 Select Account Score Metric Drivers — at what dimensions can attribute weights differ? (e.g., a weight for Employee Band may differ by region). Select the applicable dimensions.
  7. 7 Select applicable reporting dimensions for account segment movements — identify which dimensions should be tracked when accounts move between segments (e.g., year-over-year movement).
  8. 8 Click Apply / Calculate before moving on.
🚨 Rule vs. Score Precedence

Rule-Based ALWAYS overrides Score-Based when both are available for the same account. Score-based acts as a catch-all default only where no rule exists. Design your rules with this in mind — a well-designed rule set should cover your most strategically important accounts; scoring handles the rest.

Must be from the configurator, not from the generated UX app.
S4
Wallet Size Configuration — Spend Assumption Dimensions
⏱ 10 min
Objective Configure the spend assumption dimensions and reporting setup for wallet size calculation. This feeds directly into T&Q as the basis for addressable market and target allocation.
  1. 1 Navigate to the Wallet Size configuration page in the Segmentation configurator.
    [SME CONFIRMATION NEEDED: Exact configurator page name/path for Wallet Size / Spend configuration]
  2. 2 Select applicable attributes/dimensions to inform different spending assumptions — choose which dimensions drive different spend assumptions (e.g., Industry, Revenue Band). Different industries or account sizes may have different addressable market percentages.
  3. 3 Select applicable dimension to have a default addressable Market % — choose the dimension at which a default addressable market percentage is applied when no specific assumption exists.
  4. 4 Select applicable territory hierarchy level for spend index driver — select the hierarchy level at which geographic spend index adjustments are applied (e.g., region-level adjustments to account for market differences).
  5. 5 Configure any reporting setup and selection drivers applicable for wallet size analysis.
  6. 6 Click Apply / Calculate before moving on.
📝 Note

Wallet size is optional. If CloudWorks does not have reliable firmographic data to support spend calculations, skip this section. Do not over-engineer a model the customer won't trust or maintain.

Debrief Questions — Segmentation

  1. In S1, you identified both qualitative and quantitative segmentation attributes for CloudWorks. Give one example of each. For your quantitative attribute, what banded version did you need to create — and why does that banded version matter for the rule engine?
  2. In S2, you had to distinguish between Categorical Metrics and Numerical Metrics for scoring. What was the difference in how you selected them, and what does each one produce in the generated app for the admin to manage?
  3. A colleague says: "We configured score-based segmentation, so we know every account will get a segment — we don't really need rule-based." How do you respond? What is the correct relationship between rule-based and score-based segmentation?
  4. In S4, you configured spend assumption dimensions by Industry and Revenue Band. A customer asks: "Can we set different addressable market percentages for each geography as well?" What would you need to configure to support that, and where in the configurator would it apply?
  5. UCP stands for Unacceptable Customer Profile. What does it do to accounts that match its criteria? How is this different from what a learner might incorrectly assume it does?
✅ Lab Complete

When the group reconvenes, refer to the Answer Key in the Appendix to compare your decisions. There are often multiple valid approaches — the goal is to understand the reasoning behind each choice.
