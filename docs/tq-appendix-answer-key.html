T&Q Configurator Exercise — Answer Key

Reference answers for the CloudWorks Technologies configurator lab

Appendix Answer Key Facilitator Reference

📎 Appendix — Reference only. Complete the lab exercise before consulting this page.

This answer key covers all eight sections of the T&Q Configurator Exercise for CloudWorks Technologies, a fictional B2B SaaS company selling software subscriptions to mid-market and enterprise customers. The answers below reflect correct configuration choices given CloudWorks's specific scenario.

📋 CloudWorks Quick Reference

Company: CloudWorks Technologies — B2B SaaS, software subscriptions
Hierarchy: Geo → Area → Region → Sub-Region → Territory (5 tiers)
Data files: Account, Employee, Opportunity Line, Territory
Planning measures: Based on Opportunity Line revenue data
CRM: Salesforce · Integration: ADO

1.00 How to Use This Answer Key

Each section below corresponds to one page of the T&Q Configurator. For each section you'll find:

  • Correct configuration values — the specific selections and settings for CloudWorks
  • Rationale — why each choice was made given the case study data
  • Common pitfalls — mistakes participants frequently make with this section

There are often multiple defensible approaches. When your answer differs from the key, ask: "Given what I know about CloudWorks, what information would change my answer?" That's the real learning objective.

1.01 Data Hub Setup — Answer Key

Top-Level Settings

SettingAnswerWhy
Use Existing Data Hub? No CloudWorks has no existing Anaplan environment — this is a greenfield SPM deployment.
Use ADO? Yes IT team confirmed ADO readiness; ADO is the standard integration method for Polaris-based T&Q.
Use Flat File Imports? No ADO is available and confirmed — flat file is a fallback, not the target state.

Data Objects to Map

Employee Yes Employee Required for sales rep roster and territory-to-rep assignments.

Account Attributes — Enable for Calculation

Only attributes needed for reporting, rule-based assignment, or calculations should be enabled. Unused enabled attributes bloat the Data Hub.

Employee Attributes — Custom List Additions

AttributeActionWhy
Job FamilyAddDistinguishes AE, SE (overlay), and management roles for capacity and rep assignment logic.
Job CategoryAddSupports quota composition when overlays are included.

Common Pitfalls for CloudWorks

  • Enabling Geographic Mapping attributes (HQ State, Country) — CloudWorks territories are industry/segment-defined, not geo-based. Enabling these adds noise and misleads the Rule Engine setup.
  • Skipping Employee attributes — without Job Family, you cannot distinguish AEs from SEs, which breaks the overlay rep assignment flow.
  • Forgetting to enable Account Segment for Auto-Create — new accounts flowing in via ADO won't auto-generate in the segment list, causing assignment failures.
  • Adding Opportunity Line attributes to Account — Opportunity Line revenue data belongs in the Transactional Data section (Section 1.03), not as Account attributes.

Representative Example Values

Geo (L1)Americas, EMEA, APAC

Common Pitfalls for CloudWorks

  • Using only 4 tiers (copying the existing Case Study page, which shows an older 4-tier structure) — CloudWorks's current implementation uses 5 tiers. Verify against the Territory data file before committing.
  • Leaving default tier names (L1, L2, L3...) — generic names make downstream module formulas and UX labels confusing for the customer's team.
  • Not mapping attributes after clicking Apply — the "map attributes" step is a second action after naming tiers. It's easy to skip and the configurator doesn't always warn you.
  • Setting the wrong tier as the planning target — quotas are set at Territory (L5), not Sub-Region. Confusing these requires a rebuild.
1.03 Scenario Planning — Answer Key

Transaction Data Source Configuration

SettingAnswerWhy
Source data table Opportunity Line CloudWorks plans against Opportunity Line revenue data — this is the transactional fact table.
Revenue attribute Amount / Revenue The currency field on Opportunity Line representing closed-won subscription revenue.
Date field (for time routing) Close Date Close Date on the Opportunity Line determines which period the revenue belongs to.
Time scale Monthly Source data arrives monthly; the model aggregates to quarterly for planning display.
Time offset 0 No lag between opportunity close and revenue recognition for CloudWorks's subscription model.

Planning Measures

MeasureTypeSource FieldUsed For
ARR (Annual Recurring Revenue) Currency (USD) Opportunity Line → Revenue / Amount Primary planning measure — all territory targets and rep quotas
New Logos Count (Integer) Opportunity Line → New Logo Flag (Boolean → count) Secondary measure — net-new customer acquisition tracking
💡 Planning measures are very hard to change post-generation. If the customer hints at adding a third measure (e.g., Services ARR, Pipeline) during the session, add it now as inactive rather than waiting.

Forecast vs. Actuals Source

Data TypeSourceAttribute
ActualsOpportunity LineRevenue (closed-won only)
ForecastOpportunity LineExpected Revenue (open pipeline, weighted by probability)

Common Pitfalls for CloudWorks

  • Routing actuals from Account instead of Opportunity Line — Account-level actuals are a legacy pattern. For CloudWorks's subscription model, Opportunity Line is the correct source.
  • Setting time scale to Quarterly instead of Monthly — the source data granularity is monthly. Setting Quarterly here loses data fidelity; the model can still display quarterly views.
  • Missing the New Logo Flag mapping — without mapping the Boolean New Logo field to the count measure, the New Logos measure will always return zero.
1.04 Historical Data Reporting — Answer Key

Seasonality Configuration

SettingAnswerWhy
Hierarchy level for seasonality calculation Region (L3) Seasonality patterns differ between Americas and EMEA but are consistent within Regions. Territory-level seasonality would be noisy due to rep turnover.
Seasonality periods Monthly (12 periods) Monthly granularity captures intra-quarter patterns; quarterly would lose the end-of-quarter spike typical in SaaS.
Historical periods to include 2 years CloudWorks provided 2 years of actuals. Using both smooths out anomalous years.

Common Pitfalls for CloudWorks

  • Setting seasonality at Territory level — territory-level seasonality is unreliable when reps turn over; use Region or higher.
  • Using only 1 year of data — a single year may contain anomalies (CloudWorks had a spike in Q4 of one year due to a product launch). Two-year average is more stable.
  • Discussing seasonality with the customer after configuration instead of before — seasonality approach should be agreed in Session Zero. Flag this if it was skipped.
1.05 Account Value KPI — Answer Key

KPI Configuration

SettingAnswerWhy
Account attribute for Territory (for initial assignment) Territory Code The Territory file includes a Territory Code field that maps accounts to their current territory. This seeds the initial account-to-territory assignments before the Rule Engine runs.
Account Value KPI attribute Current ARR For a SaaS business, current ARR is the primary signal of account value for territory balancing decisions.
Secondary KPI attribute Potential Spend / Whitespace Potential spend captures growth opportunity — useful alongside current ARR for territory equity analysis.

Common Pitfalls for CloudWorks

  • Leaving "Account attribute for Territory" blank — this field seeds the initial rep-to-territory assignment on import. An empty field means all accounts start unassigned.
  • Using Employee Count as the KPI instead of ARR — Employee Count is a segmentation attribute, not a value signal. Always use revenue-based KPIs for territory balancing.
  • Not enabling the secondary KPI — whitespace/potential spend data is available in CloudWorks's account file; leaving it out limits the sales ops team's balancing toolkit.
1.06 Target Allocation — Answer Key

Target Entry and Cascading

SettingAnswerWhy
Target entry tier (top-down cascade starts here) Area (L2) Finance provides targets at the Area level (North America, EMEA, etc.). Entering at Geo is too high; entering at Region requires Finance to break down further than they currently do.
Time dimension for target import Annual (broken into Q) Finance provides annual targets with a Q1-Q4 seasonality split. Import as annual; the model applies seasonality to generate quarterly targets.
Over-allocation enabled? Yes CloudWorks intentionally over-assigns quotas to account for ramp, attrition, and territory vacancies.
Over-allocation maximum 115% Standard SaaS practice — 10–20% buffer. CloudWorks specified 115%.

Allocation Methods

MethodEnable?Why
Weighted by Historical ActualsYesPrimary allocation method — territories with higher historical performance receive proportionally larger targets.
Manual OverrideYesSales leaders need to adjust individual territory targets for business reasons (new territory, rep change, strategic accounts).
Pipeline AmountYesAdd Pipeline as an alternative allocation method — useful for new territories with no historical data.
Equal DistributionNoCloudWorks territories are unequal in size and maturity. Equal distribution would produce invalid targets.

Approval Workflow

SettingAnswer
Approval workflow enabled?Yes
Stage 1 approverRegional Manager
Stage 2 approverArea VP
Total approval stages2

Common Pitfalls for CloudWorks

  • Setting target entry at Geo (L1) instead of Area (L2) — Finance works at Area level. Entering targets at Geo forces Finance to submit a single global number, which they don't have.
  • Disabling Pipeline Amount as an allocation method — CloudWorks has new territories created this year with no historical actuals. Pipeline is the only basis for allocation in those territories.
  • Forgetting to set the over-allocation ceiling — leaving it blank defaults to 100% (no over-allocation), which conflicts with CloudWorks's stated intent.
  • Setting 3 approval stages when only 2 are needed — adding the Geo VP as a third approver creates unnecessary process overhead for standard quarterly target cycles.
1.07 Territory & Resource Assignment — Answer Key

Territory Assignment Settings

SettingAnswerWhy
Account assignment destination Territory (L5) Accounts are assigned to individual rep territories, not to Regions or Sub-Regions.
Sales Rep assignment destination Territory (L5) Reps are assigned at the territory level — one primary rep per territory.
First-line manager assignment destination Sub-Region (L4) First-line managers own Sub-Regions (multiple territories each).
Time dimension for quota Monthly Per the configurator guide, quota is set at the monthly level in the majority of implementations. Monthly allows for ramp schedules and mid-year adjustments.
Manual Assignment Yes Required for named enterprise accounts that Sales Ops assigns manually.
Geographic Mapping No CloudWorks territories are industry/segment-based, not geography-based.
Rule Engine Yes Mid-market accounts are assigned via rule-based automation (see Section 1.08).
Optimizer No Out of scope for this implementation. CloudWorks is not ready for optimizer; manual + rule engine covers their needs.
Split Accounts Yes Enterprise accounts with multiple divisions can span multiple territories.

Resource Assignment Settings

SettingAnswerWhy
Sales Rep level Territory (L5) AEs are assigned one-per-territory at the Territory level.
Sales Leader level Sub-Region (L4) First-line managers own Sub-Region blocks of territories.
Overlay level Region (L3) SEs (Pre-Sales Engineers) cover entire Regions — they are not assigned to individual territories.
Approval levels enabled? Yes — 2 levels Sales Ops submits → First-line Manager approves → Area VP final approval for territory changes.
Sales Rep Attribute for Territory (initial assignment seed) Territory Code (on Employee record) The Employee data file contains a Territory Code field that maps each rep to their current territory. This seeds the initial state before ADO sync runs.

Common Pitfalls for CloudWorks

  • Assigning overlay SEs at Territory level — SEs cover Regions, not individual territories. Territory-level assignment creates ~220 overlay entries instead of ~6, causing UX and calculation issues.
  • Leaving "Sales Rep Attribute for Territory" blank — without this seed, all reps start unassigned and the initial data load produces an empty assignment state.
  • Setting quota time dimension to Quarterly — the exercise guidance says Quarterly for granularity display, but the configuration guide recommends Monthly for quota. Monthly allows ramp curves; Quarterly does not.
  • Enabling Geographic Mapping — CloudWorks explicitly stated territories are NOT geography-defined. Enabling this creates a zip-code / state mapping process that doesn't apply.
1.08 Rule-Based Assignment — Answer Key

Qualifying Attributes (Which accounts are eligible for rule assignment?)

AttributeInclude?Why
Account Segment Yes Only Commercial and SMB segments are eligible for rule-based assignment. Strategic and Named accounts are manually assigned.
Account Type Yes Distinguishes pool accounts (eligible for rules) from named accounts (manually assigned).
Employee Count Yes Banded employee count separates mid-market (<1,000) from enterprise (1,000+), which drives territory type routing.
HQ Country No Geographic rules are disabled for CloudWorks — territories are not defined by location.
📝 Note

Selected attributes must be in list format line items (not free text or numeric). Ensure Account Segment and Account Type are configured as list-type attributes in Data Hub Setup — if they're text fields, the rule engine cannot use them.

Rule Definition Attributes (Used to define territory parameters)

AttributeInclude?Why
Industry Yes CloudWorks has industry-specialized territories (e.g., FinServ territory, Healthcare territory). Industry drives which territory a mid-market account routes to.
Account Segment Yes Commercial vs. SMB accounts route to different territory types even within the same region.
Employee Count Band Yes Banded employee count (e.g., 100–249, 250–499) further refines territory routing for mid-market accounts.

Example Rule for CloudWorks

Rule: Industry = "Technology" AND Account Segment = "Commercial" AND Employee Count Band = "250–999"
→ Assign to: US-West / Bay Area / SF-Commercial-Tech Territory

Rule: Industry = "Financial Services" AND Account Segment = "Commercial"
→ Assign to: US-East / Greater NY / NY-FinServ Territory

Common Pitfalls for CloudWorks

  • Using text-type attributes instead of list-type — the Rule Engine requires list-format line items. If Industry is stored as free text in Salesforce, it must be standardized to a list in Data Hub before rules can reference it.
  • Trying to use geographic attributes (State, Country) in rules — CloudWorks territory is NOT geographic. Rules based on HQ State will not match CloudWorks's territory structure.
  • Including Strategic/Named accounts in the qualifying set — these are manually assigned. Including them in the rule-eligible pool creates conflicts between manual assignments and rule outputs.
1.09 Optimizer — Answer Key

Optimizer Configuration Decision

SettingAnswer
Enable Optimizer? No — Out of scope for CloudWorks Phase 1
💡 Rationale: CloudWorks explicitly stated the Optimizer is not in scope for this implementation. They're moving from Excel — introducing Rule Engine + Manual assignment is already a significant change. Optimizer is typically Phase 2 after data quality is validated.

What You Would Configure (If Optimizer Were Enabled)

For reference — if CloudWorks asked for Optimizer in a future phase:

SettingRecommended ValueWhy
Objective (primary) ARR (balance revenue potential across territories) The primary planning measure should be the optimization objective for equitable territory design.
Objective (secondary) Pipeline Amount Pipeline is the forward-looking signal; including it balances current ARR with growth opportunity.
Geography Constraints level Sub-Region (L4) Optimizer constraints should apply at Sub-Region level — territories within a Sub-Region can be rebalanced, but accounts shouldn't move across Sub-Regions without manager approval.
Mix Constraints attribute Account Segment Ensures Optimizer doesn't create a territory with only SMB accounts or only Strategic accounts — each territory should have a healthy segment mix.

Common Pitfalls (General)

  • Enabling Optimizer without confirmed data quality — Optimizer requires clean, complete account-level actuals. Garbage in, garbage out. Always validate data quality before activating.
  • Setting Geography Constraints at Territory level — too granular; locks the optimizer into micro-moves that produce minimal improvement.
  • Not including Pipeline as a secondary objective — for B2B SaaS with long sales cycles, historical ARR alone is a lagging indicator. Pipeline is critical for balancing growth potential.

Facilitator Debrief Notes

When reviewing answers with the group, focus discussion on these decision points where participants most commonly diverge:

🔥 Highest-Variance Decisions

  • Tier count (1.02): Some participants choose 4 tiers based on the existing Case Study page (which shows an older structure). The exercise data shows 5 tiers — this is intentional to test whether they verify against the data file.
  • Target entry level (1.06): Most common wrong answer is Geo (L1). Finance works at Area (L2). Dig into how they discovered Finance's target granularity.
  • Quota time dimension (1.07): Monthly vs. Quarterly is a genuine debate. Monthly is technically correct; either is defensible if the participant can explain why.

🎯 Key Teaching Moments

  • Data Hub drives everything: Attributes not enabled here cannot be used anywhere else. Decisions in 1.01 cascade through every other section.
  • Optimizer is not a Phase 1 item: Push back if any participant says "we'd enable it anyway." Build the habit of right-sizing scope.
  • List-type vs. text attributes: The Rule Engine limitation (list format required) is a real implementation gotcha that catches many delivery teams.
  • Post-generation is expensive to fix: Planning measures, hierarchy depth, and approval workflow stages are all very difficult to change after model generation.
