Anaplan Way for Apps

The 7-step T&Q implementation methodology

Section 2 45 min Methodology

The Anaplan Way for Apps is the prescribed delivery methodology for implementing RPM applications, including T&Q. Unlike a full custom Anaplan build, the Apps delivery motion is structured around the configurator tool — which means the sequence of activities, the timing of key milestones, and the division of labor between Anaplan specialists and the delivery team are all well-defined. Follow this methodology closely; deviation is the primary cause of rework and project overruns.

Implementation Phases

1

Session Zero — Data Readiness

Session Zero is a pre-kickoff engagement focused entirely on data. Before a single configurator setting is touched, you need to understand the customer's data landscape: what systems they use, what data they have, what format it's in, and how clean it is.

Key activities:

  • Share the Data Prep Workbook with the customer as early as possible — ideally before the SOW is signed
  • Walk through the workbook requirements: account hierarchy, territory hierarchy, rep list, historical actuals, target file
  • Identify data gaps, inconsistencies, and transformation needs
  • Confirm which systems of record will feed T&Q (CRM, ERP, HR, manual)
  • Validate sample data — you need real rows, not just schema definitions
🚫 Hard Rule

Do not kick off the project without sample data. If the customer can't provide even a representative sample of accounts, territories, and reps before kickoff, the configuration sessions will be guesswork. Push back on this. Projects that start without data always get stuck later.

2

Kickoff

The formal project kickoff with the customer team. Use the Anaplan-provided kickoff template deck (available in the enablement portal). This is not a generic project kickoff — it should orient the customer specifically to the T&Q app and what they're about to go through.

Key activities:

  • Review the project plan and milestone schedule
  • Introduce the configurator and what it produces
  • Align on the app scope (which pillars, which features)
  • Discuss yellow tabs vs. red tabs in the configurator:
    • Yellow tabs — configurations that can be adjusted post-generation with moderate effort
    • Red tabs — configurations that are very difficult or impossible to change after generation; must be finalized during the configuration sessions
  • Set expectations about the generation process (timing, errors, what comes next)
3

Base App Walkthrough

Before configuration sessions begin, walk the customer through the base T&Q app at a high level. This is a demo of what they're going to get — it sets context, manages expectations, and often surfaces requirements that weren't captured in discovery.

Key activities:

  • Demo the spoke app end-to-end (or show the reference video)
  • Highlight the major functional areas: Territory Management, Target Setup, Quota Setup, Account Assignment, Administration
  • Point out what is and isn't configurable via the configurator
  • Capture questions and requirements that arise — feed these into configuration sessions
💡 Tip

Use this walkthrough to "anchor" the customer's expectations on base functionality before they start asking for extensions. The more familiar they are with what the app already does, the less time you spend building net-new things that are already there.

4

Configuration Sessions (1–2 Weeks)

The core of the pre-generation work. These are structured working sessions — typically 2–4 hours each, held daily or every other day — where you walk through the configurator with the customer's decision-makers and complete every setting.

Key activities:

  • Work through configurator pages 1–14 systematically
  • Bring the right customer stakeholders to each session (Sales Ops for hierarchy/planning measures, Finance for targets, IT for data integration)
  • Document every decision — especially anything that deviates from default settings
  • Validate decisions against the sample data (e.g., how many hierarchy tiers does their data actually have?)
  • Flag red-tab decisions for extra review and sign-off
⚠ Warning

Don't let configuration sessions drag past 2 weeks. Scope creep and analysis paralysis are the main risks. Keep decisions moving. If a customer can't decide on a setting, document the tradeoffs, make a recommendation, and get sign-off. You can't generate an app on a "maybe."

5

App Generation

The configurator generates the T&Q application. This is handled by Chan/John (the App Engineering team) — do not run generation yourself without their involvement. Generation is a complex, time-sensitive process that should be treated as a controlled event.

What happens during generation:

  • The configurator reads all completed settings and generates approximately ~8,000 Anaplan objects (modules, line items, lists, dashboards, actions, etc.)
  • Generation runs for approximately 90 minutes to 2 hours
  • 0–200 errors after generation is acceptable — this is normal and expected. Errors above 200 warrant investigation
  • Generation should be run in a clean, dedicated workspace — not a workspace with existing models
📝 Note

The generation errors you see post-run are typically broken formula references due to conditional module generation — some modules only exist if certain features are enabled. The post-gen checklist addresses how to resolve these.

6

Model Readiness (1–2 Weeks)

After generation, the app exists but is not ready for the customer. Model Readiness is the period of post-gen work required to get the app to a demonstrable, data-loaded state. With focused effort, this is approximately 1–2 days of actual work, spread over 1–2 weeks to allow for customer data prep.

Key activities:

  • Work through the post-gen checklist (see the Post-Generation Steps section)
  • Resolve generation errors and broken formula references
  • Load customer data (accounts, territories, reps, historical actuals, initial targets)
  • Validate data loads — check rollups, check hierarchy integrity
  • Configure ADO pipelines (or existing Data Hub) for ongoing data feeds
  • Demo the loaded app back to the customer — this is the Model Readiness milestone
7

Extension Design

With the base app live and the customer seeing their own data, now is the right time to revisit the extension list. Customers almost always have a wishlist of custom functionality. Extension Design is about separating what the base app already handles from what genuinely requires new build.

Key insight: When you show customers the loaded base app, 1/3 to 1/2 of their planned extensions often turn out to be unnecessary — the base app already covers the need, or the customer's requirement was based on a misunderstanding of how Excel worked. Always anchor extension scoping on a demo of the live app, not a requirements list written before they saw the app.

Key activities:

  • Review the extension wishlist against the live app
  • For each item: does the base app do this? Can it be configured? Does it require net-new build?
  • Size and prioritize remaining extensions (see Common Extensions section)
  • Plan extension build into the project timeline

Configure vs. Extend

🔧 Key Concept

Configure: Choices made in the configurator that shape the generated app. No model building required. Examples: number of hierarchy tiers, which planning measures are included, whether the approval workflow is enabled.

Extend: Net-new model building added on top of the generated baseline. Examples: custom dashboards, additional approval stages, non-standard quota components, integrations to non-standard systems.

The rule of thumb: exhaust configuration options before scoping extensions. Extensions are expensive to build and maintain — especially across app upgrades.

Two Customer Types

You will encounter two broad customer profiles in T&Q engagements. Understanding which type you're dealing with shapes how you run the project:

🆕 Net-New (Greenfield) Customer

No existing Anaplan for SPM. Coming from spreadsheets, a legacy tool, or a custom-built solution. The configurator generates their first T&Q app from scratch.

  • Data migration/transformation is the biggest risk
  • Change management is critical — users have no Anaplan muscle memory
  • More likely to have a long extension wishlist
  • Session Zero investment is highest here

🔄 Upgrade (V2 → V3) Customer

Existing T&Q on an older version (typically V2). Upgrading to V3 to access new functionality, improved performance, or ADO integration.

  • Has existing data structures that may not map cleanly to V3
  • May have existing extensions that need to be re-scoped for V3
  • Territory hierarchy structure changes between V2 and V3 (flat file vs. per-tier views)
  • User adoption is usually easier — team already knows Anaplan
⚠ Upgrade Path Warning

V2 → V3 upgrades are not automatic. The V3 app is a new generation — there is no in-place upgrade path. You generate a new V3 app and migrate data. Any custom extensions built on V2 will need to be re-evaluated and potentially rebuilt on V3. Set this expectation with customers early. The upgrade should be scoped as a new engagement, not a "version bump."
