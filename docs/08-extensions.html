Common T&Q Extensions

Scoping, sizing, and best practices for extending the T&Q application

📖 Tell When and How to Extend You'll learn which extensions are most common, how to scope and size them, and how to protect the upgrade path.
👁 Show Extension Examples Instructor walks through two or three real extension examples — how they were scoped, built, and documented.
✅ Do Scope an Extension Given the CloudWorks scenario, identify one extension the customer would likely need. Scope it: what does it do, how long, what risk?

Configure vs. Extend — Decision Framework

Before writing a single line of custom logic, work through this decision tree with the customer. Extensions are expensive to build and to maintain across upgrades. Exhaust configuration options first.

Does the base app include this capability at all?
No
Extension required — scope it, size it, document it from the start.
Yes
Is the customer's requirement a configuration setting in the Configurator?
Yes
Configure it. Go back to the Configurator — this is a generation-time decision.
No
Can the customer change their process to match what the base app provides?
Yes
Recommend the process change. This is a consulting conversation, not a build conversation. Changing the process protects their upgrade path forever.
No
Is this a small calibration (label rename, add an export action, minor formula tweak)?
Yes
Post-gen configuration — handle it during Model Readiness. Not a project extension scope item.
No
Extension required — but build it cleanly: net-new modules, EXT_ naming prefix, full documentation. Never modify base app line items.
💡 The Golden Rule

When a customer shows you a custom spreadsheet column and says "we need this in Anaplan" — your first question is always: "Does the base app already provide this?" Show them the base app page that covers it before agreeing to build anything. This conversation alone eliminates a large fraction of extensions on every project.

No Zero-Effort Implementations
Even when the base app covers a use case, there is always calibration effort to fit it to a specific customer. The "pure configure, no extensions" path has happened exactly once — for a customer with no existing process who took whatever the app gave them. For everyone else: expect extensions.

Extension Best Practices

How you build extensions matters as much as what you build. Follow these principles to protect the upgrade path and reduce long-term maintenance burden:

  1. Document everything. Every extension should be noted — module name, purpose, what base app component it relates to. Use a naming convention (e.g., prefix extension line items with "EXT_") so they're easy to identify.
  2. Keep extensions separate from baseline. Don't modify base app line items. Create net-new line items that reference or extend the base. When an upgrade ships, your net-new items are untouched.
  3. Create net-new line items instead of modifying base formulas. If you need different logic than what the base provides: create a new line item with your formula, and reference that new item downstream. The base item remains intact and will be correctly overwritten on upgrade.
  4. Anticipate extensions during discovery. Use the extension list and discovery questions below to identify likely extensions before the project starts. Scope them explicitly — don't let them emerge as surprises mid-build.
  5. Challenge every extension request. One-third to one-half of initially identified extensions can often be addressed using base app functionality — either used differently or explained to the customer as a better practice. Always validate whether the base app can handle it before building net-new.

Common Extensions by Functional Area

Area Extension Size Notes
Resource & Quota Position Management M Insert a "position" entity between territory and employee. Common for enterprise customers with headcount planning tied to roles rather than individuals. Connects well to Capacity Planning output.
Quota Currency Conversion S–M Required when customer sets targets in USD but reps operate in local currencies. Applies conversion rate to quota reporting and acknowledgement output.
Quota Custom Proration Logic S–M When customer's proration rules differ from the system default (e.g., hire date rounding to quarter vs. month, specific LOA rules, different handling for promotions).
Quota Weighted Allocation Blend S Allow admins to set a percentage weight for each allocation method (e.g., 60% prior year, 40% pipeline). Just a line item and subset reference — small effort, high value.
Quota Quota Letter / Acknowledgement M–L Basic: rep-specific page showing their quota composition for review. Advanced: DocuSign connector integration for a signed acknowledgement. DocuSign connector maps quota values to a template and captures a digital signature.
Quota Per-Role Raw/Discrete Selection S Base app applies raw vs. discrete quota universally per level. Extension allows selecting raw or discrete per role per quota type, giving managers more nuanced control.
Account Assignment Account Ownership Export S The export module exists out of the box (account × territory assignment); it just needs an export action added. Required to sync account ownership back to CRM at end of planning.
Account Assignment Split Account Management M Allows an account to be assigned to multiple territories with defined split percentages. Coming in V3 natively; requires extension in V2.
Account Assignment Geomapping Enhancement M Requires a cleaned-up geography list with correct boundary polygons for the mapping widget. Customer must have latitude/longitude on account records. Source available from Anaplan's geography list resource.
Account Assignment Boomerang Account/Resource M V3's time-intersection model loses day-level precision for assignments. If a rep leaves and returns within the same month, or an account moves mid-month, extension needed to handle day-level effective dating.
Territory Child Follows Parent S When reparenting a territory, all child territories and their account assignments move with it. Base app only moves the selected territory. Simple reference update extension.
Target Target Setting by Exception M Some managers want to set targets for only select territories rather than running the full allocation across all. Requires a flag/selector layer and filtered allocation logic.
Reporting Custom Manager Dashboards S–XL Almost always required. Base dashboards are conversation starters. Final dashboards are customer-specific, incorporating their preferred KPIs, hierarchy breakdowns, and visual layout.
Admin Parsed Data Load Actions S Replace the generated "load everything" bulk action with per-object actions (Account load, Employee load, etc.). Improves error management and gives admins control over what to refresh and when.
Data Custom Data Objects S per object Additional hierarchical lists or flat data feeds beyond the standard objects (accounts, employees, opportunities, orders, territories). Common examples: Business Unit hierarchy, Partner hierarchy, custom scoring data.

Upgrade Path for Extensions

When a new version of the T&Q app ships, here's how to manage the upgrade with extensions in place:

  1. Review the release notes. Understand what modules, line items, and UX pages are being updated in the new version.
  2. For extensions on untouched base modules: No action needed. Your net-new modules are unaffected by the upgrade.
  3. For base line items you needed to reference differently: Compare the new version's formula against what was there before. If the logic improved, update your extension's reference. If it's the same, no action needed.
  4. For extensions that added new modules: No impact from upgrade. They exist outside the base app's module set.
  5. Use the template app to preview changes. Before upgrading a customer, spin up the new template version and compare it side-by-side with the current customer model. Identify what's different, then plan the merge.
  6. Current reality: Until Anaplan's Path/AAF framework matures, upgrades are essentially a faster re-implementation. Well-documented, cleanly separated extensions make this significantly faster — you know exactly what to move over.
Modified Base App Line Items
If you modified a base app line item's formula (not best practice, but it happens), that modification will be overwritten when the app is upgraded. This is why creating net-new line items is critical. You can reference a base item in your extension — just don't edit the base item itself.

Pre-Project Discovery Questions

Use these questions before or at project kickoff to assess T&Q fit and identify likely extensions early. ⭐ marks the highest-priority questions.

Territory Structure

  • ⭐ Do you have a defined territory hierarchy today, or do your reps "own" accounts directly?
  • ⭐ How many levels does your territory hierarchy have? Does this differ by region (e.g., 4 levels in Americas, 3 in EMEA)?
  • Is your territory hierarchy finalized and approved, or still under discussion?
  • Do accounts move between territories mid-year? How often?
  • Do you have global accounts or named accounts with special coverage requirements?

Quota & Target Setting

  • ⭐ What planning measures do you set targets against? (e.g., Bookings, New ARR, Renewals, Product-specific). How many?
  • ⭐ At what level does Finance provide initial targets — region, area, or district?
  • ⭐ What time granularity are targets provided at — annual or quarterly?
  • Do you set targets by product or product family? For how many products?
  • How do you handle over-allocation? Is there a standard uplift percentage per level?
  • Do you have a formal target approval workflow today? Who approves at each level?
  • Do you need quota acknowledgement (rep sign-off)? Is DocuSign in scope?

Rep Coverage & Assignments

  • ⭐ How do you handle in-year changes? (New hires, attrition, promotions, LOA, account transfers)
  • Do you use overlay roles (product specialists, solution engineers, customer success)? At which hierarchy levels?
  • Do you use role-based quotas (all reps with the same role in the same region share the same quota)?
  • Do any reps cover multiple territories simultaneously?

Data & Integration

  • ⭐ What is your current data source for accounts, opportunities, and territory hierarchy?
  • Do you have an existing Anaplan COE with a Data Hub? Or is this greenfield?
  • Do your account records include latitude and longitude attributes?
  • Do you have unique identifiers on all key objects (accounts, employees, territories)?
  • How frequently does account data need to refresh in the planning tool?

Process Maturity & Fit

  • ⭐ Are stakeholders available and empowered to make decisions during the project? What is their availability (hours/week)?
  • Do you have a designated COE owner or admin who will maintain the app post-go-live?
  • How do you currently manage T&Q planning today (Excel, legacy system, other Anaplan model)?
  • Is there organizational appetite to adopt Anaplan best practices, or do you need the tool to mirror your existing process exactly?
  • What does your ICM process look like today? (Sets up the future ICM expansion conversation)

  • Write a one-sentence scope statement: "We will build X to support Y, using Z approach."

  • What naming convention and documentation would you use to protect the upgrade path?
