Post-Generation Steps

From raw generation to Model Readiness — the complete post-gen checklist

Section 5 45 min Reference

After the T&Q app is generated, you have a structurally complete application — but it is not yet ready for the customer. There is a defined set of post-generation steps that must be completed before the app can be considered "Model Ready" and presented to the customer with data. With focused effort, the actual hands-on work is approximately 1–2 days, though it is typically spread over 1–2 weeks to allow for data preparation on the customer side.

The goal of the Model Readiness phase is to deliver a live demo of the app with the customer's own data loaded, so they can validate the configuration choices and sign off before building begins.

⚠ Known Issues

The following known issues occur regularly after generation and should be expected. Do not escalate these as bugs unless they persist after the documented fix:

  • Source model bug: Some formula references in generated modules point to the configurator source model rather than the generated app. These must be manually updated in the affected line items. The App Engineering team maintains a log of which line items are typically affected for each version.
  • Save view mappings: Dashboard saved views are not always correctly mapped after generation. Check every dashboard page and re-save any view that shows incorrect filters or formatting.
  • Broken formula references: Modules that are conditionally generated (only exist when certain features are enabled) sometimes have forward references from always-generated modules. These show as formula errors. Work through the error log systematically — most are one-line fixes.

Post-Generation Checklist

Work through this checklist in order. Each category contains sub-steps — see the full printable checklist (distributed as a separate document) for the complete item-level list.

1. Generation Validation

  • Review the generation error log — count total errors (0–200 is acceptable)
  • Categorize errors: formula errors vs. structural errors vs. data errors
  • Identify and log any errors above the expected count for the configuration
  • Confirm all expected modules, lists, and dashboards were created
  • Confirm workspace size post-generation — check for any workspace overage warnings

2. Formula & Reference Fixes

  • Open the model errors panel — work through every error
  • Fix source model references (broken links to configurator source model)
  • Fix forward formula references to conditionally-generated modules
  • Verify all line items in core modules (Target Allocation, Quota Calculation, Resource Assignment) are formula-correct
  • Re-run any failed process actions to verify they complete cleanly

3. Dashboard & View Cleanup

  • Open every dashboard page in the spoke app
  • Check saved view filters — re-save any that are blank or incorrect
  • Verify pivot table headers are correct (not showing "undefined" or configurator placeholders)
  • Check conditional formatting rules are firing correctly
  • Verify navigation links between dashboard pages work correctly

4. Data Hub Setup & Initial Data Load

  • Load Territory hierarchy (all tiers) — verify rollup integrity
  • Load Account list — check for duplicate IDs, missing mandatory fields
  • Load Resource (rep) list — verify territory assignments are populated
  • Load Account-to-territory mapping (if using pre-assigned mapping)
  • Run the "Validate Data Hub" action and resolve any errors
  • Spot-check: select 3–5 territories and verify accounts, reps, and rollups look correct

5. Historical Actuals Load

  • Format historical actuals data to match the import template
  • Load actuals for each historical period (Year 1, Year 2)
  • Verify actuals roll up correctly to territory → region → theater → global
  • Compare total actuals in Anaplan to source system total — must reconcile to within 0.5%
  • Check for orphaned actuals (actuals with no matching account or territory)

6. Target Import & Allocation Validation

  • Import Finance targets at the configured entry tier
  • Run the allocation process down the hierarchy
  • Verify allocation totals match Finance targets (before over-allocation buffer)
  • Spot-check 3–5 territories: allocated target vs. manual expectation
  • Test the over-allocation warning — verify it fires at the configured threshold

7. Quota Calculation Validation

  • Assign test resources to territories (use test reps, not real reps)
  • Run the quota calculation process
  • Verify quota = territory target × assignment % × ramp factor
  • Test ramp scenarios: 100% assignment, 50% split, new hire ramp
  • Verify quota totals roll up correctly

8. Integration & ADO Setup

  • Configure ADO pipelines for each data source (accounts, actuals, reps)
  • Run a test data refresh end-to-end
  • Verify incremental update behavior (new accounts added, removed accounts handled)
  • Set up and test scheduling for ongoing data refreshes
  • Document the ADO pipeline architecture for handoff to customer IT

Model Readiness Goal

The Model Readiness milestone is a working session with the customer (typically the Sales Ops lead and at least one Sales manager) where you demonstrate the app with their data. The goal is:

  1. Customer sees their actual territories, accounts, reps, and targets in the app
  2. Customer can navigate the spoke app independently for the major workflows
  3. Customer signs off that the configuration reflects their requirements
  4. Any remaining configuration changes are documented as change requests
  5. Project proceeds to UAT or extension development
💡 Tip

Don't try to make the app "perfect" before the Model Readiness demo. The goal is to show enough working functionality with real data that the customer can validate. Cosmetic issues, placeholder labels, and secondary dashboards can be cleaned up during UAT. Getting to a live demo quickly builds customer confidence and surfaces real feedback.

V3 Note

📝 V3 Specifics

The V3 app generates significantly more dashboards and modules than V2. The post-gen checklist above is calibrated for V3. If you are working on a V2 → V3 upgrade, note that:

  • V3 uses per-tier hierarchy views (not a flat territory list) — the data load format is different from V2
  • V3 ADO integration replaces V2's flat-file Data Hub approach — IT will need separate setup support
  • V3 has additional dashboard categories that don't exist in V2 — plan extra time for dashboard validation
  • V2 extensions do NOT automatically transfer to V3 — each extension must be re-evaluated
