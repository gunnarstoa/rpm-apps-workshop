Seg & Capacity Overview

Segmentation, Scoring & Go-to-Market Capacity Planning

About This Module

This module builds on the T&Q foundation from the previous module. In this module you'll go deeper into two additional SPM apps: Account Segmentation & Scoring and Go-to-Market Capacity Planning. Both apps share the same configurator entry point but generate separate spoke models — you'll see how they fit together and how they feed back into T&Q.

Learning Objectives

  1. Understand the Segmentation & Scoring configurator and its key configuration decisions
  2. Understand the Capacity Planning configurator and the dimensionality tradeoffs involved
  3. Walk a customer through the Segmentation app end-to-end
  4. Walk a customer through the Capacity app end-to-end — including tops-down and bottoms-up planning
  5. Understand the Sales Forecasting app (just launched) and roadmap updates for T&Q V3, Segmentation V2, and Capacity V2

Key Relationship Between Apps

These three apps don't operate in isolation — they feed each other. Understanding the data flows between them is critical to positioning the full SPM suite.

  • Segmentation → T&Q: Account segments drive territory carving. Which accounts are Strategic vs. SMB determines how territories should be carved, sized, and assigned.
  • Segmentation → Capacity: Account mix changes drive headcount needs. If your re-segmentation exercise moves 500 accounts from Commercial to SMB, you need fewer Commercial AEs and more SMB AEs.
  • Capacity → T&Q: TPH (Territories Per Headcount) counts from Capacity inform territory coverage decisions in T&Q.
Shared Configurator: Both Segmentation and Capacity share the same configurator app — they're just different sections within it. You configure both in a single set of sessions with the customer, and the data setup is shared between them.
⚠ Important: These Connections Are Not Automatic

The data flows between Segmentation, Capacity, and T&Q described above are not automatic today. Moving data between apps requires a manual export/import step or a custom ADO pipeline. Native app-to-app connections are on the roadmap but not yet available. Plan for this in your project timelines — it's a meaningful implementation step, not a checkbox.

Inter-App Data Flow

Here's how data moves through the three apps in a complete SPM implementation:

🎯
Segmentation & Scoring
Classifies every account into a segment (Strategic, Named, Commercial, SMB)
Account segments
export → T&Q import
🗺
Territory & Quota (T&Q)
Uses segments for territory carving, rule engine, and account assignment
Territory structure
export → Capacity import
📊
Capacity Planning
Uses account mix + territory structure to calculate headcount needs

Segmentation → T&Q

What moves: Account segment attribute per account

How: Export from Segmentation app → import into T&Q Data Hub as an account attribute

Why it matters: Territory carving and Rule Engine logic in T&Q can reference the segment. "All Strategic accounts in EMEA → Enterprise Territory" only works if the segment is loaded.

Timing: Run after each segmentation exercise, before territory planning begins

Segmentation → Capacity

What moves: Account counts per segment per geography

How: Export from Segmentation → import into Capacity coverage module

Why it matters: Coverage ratios in Capacity (1 AE per N accounts) depend on knowing how many accounts are in each segment. Change the segmentation → change the required headcount.

Timing: Run after finalizing segmentation, before running headcount models

Capacity → T&Q

What moves: Territories-per-headcount (TPH) recommendations

How: Manual reference — planners use the Capacity output to inform territory structure decisions in T&Q

Why it matters: If Capacity says you need 12 Commercial AEs in US-West, your T&Q territory structure needs 12 Commercial territories in US-West.

Timing: Inform territory design before finalizing the T&Q hierarchy

Ideal Sequence

  1. Run Segmentation → finalize account segments
  2. Export segments → load into Capacity coverage
  3. Run Capacity → determine headcount needs per role/geo
  4. Use headcount model to inform T&Q territory structure
  5. Export segments → load into T&Q for rule engine + carving
  6. Run T&Q territory planning with fully-informed structure
