Go-to-Market Capacity Planning Configurator

Key configuration decisions for the Capacity Planning app

📖 Tell Capacity Configuration You'll learn the key decisions in the Capacity configurator — role hierarchy, coverage assumptions, and productivity modeling.
👁 Show Facilitator Demo Instructor walks through the Capacity configurator, highlighting the tops-down vs bottoms-up planning tradeoffs.
✅ Do Capacity Lab Complete the Capacity section of the CloudWorks configurator exercise on the lab page.

The Capacity configurator is in the same app as Segmentation — scroll down to the Capacity section. Same principle as all configurators: decisions here define dimensionality that is very painful to change post-generation. Capacity configurator = ~0.5× complexity vs. T&Q (~2 days with a prepared customer vs. ~1 week for T&Q).

Overview of Capacity Configuration Goals

  • Define the sales role / job family hierarchy structure
  • Configure coverage assumptions (accounts per role ratio)
  • Set up productivity modeling approach
  • Configure tops-down and bottoms-up planning levels
Shared data setup: If you're configuring both Segmentation and Capacity in the same sessions, the data setup is shared — you only need to configure accounts, employees, and opportunities once.

Page: Role / Job Family Hierarchy

Define the sales role hierarchy — the structure that determines how headcount is planned and reported.

  • Up to 3 levels: Job Family Group → Job Family → Sales Role
  • Key question: how many levels do you actually need? More levels = more granular planning, but significantly more maintenance overhead
  • Role grade: Optional additional dimension (e.g., P1–P5). Adds substantial complexity — only enable if the customer genuinely needs grade-level headcount planning
Challenge role grade hard before enabling it. If you configure role grade and the customer doesn't really need it, you've added substantial complexity with no value. Every additional dimension multiplies the number of rows in every coverage, productivity, and hiring table. Challenge this requirement rigorously before enabling it.

Page: Coverage & Workload

Define how roles cover accounts — the fundamental input for calculating how many people you need.

  • Configure at what level of geography you manage coverage assumptions (e.g., all Americas at once, or per sub-region)
  • Direct sellers — account-based coverage: 1 AE per N accounts
  • Overlay / support roles — derived coverage: 1 generalist per N AEs, 1 manager per N AEs
  • Segment mapping: Which roles cover which account segments (e.g., Strategic AEs cover Strategic + Named accounts only)
  • Hours-based planning: Optional workload model for non-revenue roles (e.g., 1 specialist handles 32 hours/week of customer requests). Rarely used — revenue-based coverage is the standard approach.

Page: Revenue Target

Define the dimensionality of revenue targets — how Finance will provide the numbers that drive tops-down planning.

  • At what level of detail does Finance provide revenue targets? (By region? By sub-region? By sub-region + account segment?)
  • What dimensionality? By sub-region only, or by sub-region AND account segment?
Market dimensions pitfall: The "market dimensions" input (revenue target by account segment) exists in the configurator, but it does not automatically tie the productivity of roles in that segment to the target. This requires post-generation work to route properly. Know this before committing to that dimensionality with a customer — it will add implementation effort that isn't visible in the configurator.

Page: Tops-Down Planning

Define the guardrail structure for finance-driven headcount planning.

  • At what level of the geo hierarchy does tops-down target setting begin? (Usually the top — all Americas, or global)
  • Tops-down sets the NTE (Not to Exceed) guardrails that constrain bottoms-up planning at every level below
  • Scenario planning: how many scenarios do you want to model? (Typically 2–3: conservative, base, optimistic)

Page: Bottoms-Up Planning

Define the tactical hiring planning layer — where planners actually enter month-by-month hire plans.

  • At what level are bottoms-up hiring plans entered? (Typically territory or leaf level)
  • Approval levels: once submitted, at what level does the plan get approved? (e.g., submitted at sub-region, approved at region)
  • OWP connection: Optional import of data from Anaplan Workforce Planning — ramping profiles, historical headcount data. Define at what level this data comes in if using OWP.
⚠ Critical Dimensionality Warning

Dimensionality decisions in the Capacity configurator are extremely difficult to change post-generation. If you configure cost assumptions at the region level and later decide you need sub-region, you will need to disconnect and reconnect many formulaic references — or regenerate and lose any custom work.

Be thorough and deliberate with your customer. Validate every dimensionality choice before generating. This is the #1 source of rework on Capacity implementations.

Timing Guidance

Configurator Estimated Time Notes
T&Q ~1 week Full data setup + all config pages
Segmentation ~2–3 hours Data setup shared with Capacity
Capacity ~2 days Data setup shared with Segmentation

If doing both Segmentation and Capacity together, the data setup is shared — you don't repeat it. Both configurators live in the same app, so you can run them in the same set of customer sessions without switching context.
