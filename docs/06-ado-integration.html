Data Integration & ADO

Anaplan Data Orchestrator — Current State and the V3 Transition

📖 Tell V2 vs V3 Data Architecture You'll learn how the data layer is changing with V3, what ADO replaces, and how to advise customers on timing their upgrade.
👁 Show Architecture Diagram Instructor walks through the data flow diagrams for both V2 (Data Hub) and V3 (ADO) architectures side by side.
✅ Do Customer Scenario Given the CloudWorks scenario, decide: V2 Data Hub or V3 ADO? Justify your answer to the group.

Current State vs. Future State

T&Q V2 uses a Data Hub model as the data layer between source systems and the spoke model. Starting May 2026, all three SPM apps (T&Q, Capacity Planning, Segmentation) move to Anaplan Data Orchestrator (ADO) — eliminating the Data Hub model entirely.

Aspect V2 (Current) V3 (May 2026+)
Data layer Data Hub model ADO (no Data Hub model)
Data flow direction Pull from sources into Data Hub; Hub pushes to Spoke ADO pushes directly to Spoke
Scheduling External ETL or manual ADO Workflows (included with app)
Workspace cost Data Hub consumes workspace size ADO is a separate entity — no model size cost
Territory file format One flat file (all tiers in columns) Separate source view per hierarchy tier
Access requirement Standard model access Requires Workflow entitlement (included with app)
Large transactional data Stored in Data Hub model Stored in ADO — doesn't bloat spoke model

What is ADO?

Anaplan Data Orchestrator is a separate entity — not a model. It functions as a central data repository and transformation layer. Think of it like a data warehouse that exists alongside your Anaplan workspace.

Key Differences from Data Hub

  • Push vs. Pull architecture: In V2, each model pulls data from the Data Hub. With ADO, data is pushed from ADO to all connected models simultaneously — you schedule once, all models update.
  • No workspace size cost: ADO doesn't consume Anaplan workspace. Customers with large transactional datasets (e.g., 3 years of opportunity data) won't blow up their Polaris spoke model.
  • Native transformation logic: ADO has its own transformation layer (different from Data Hub formula approach). Data shaping happens before it reaches the spoke.
  • Scheduling via Workflows: ADO uses Workflows to schedule data pushes. This is why the Workflow entitlement is bundled with app subscriptions.
  • Multi-model push: If a customer has T&Q, Capacity, and Segmentation, one ADO push can feed all three — no need to manage separate import processes per model.
Polaris + Large Data
A Siemens Opportunity Data module with 3 years historical + future data reached 20GB. ADO prevents this kind of bloat by keeping transactional data out of the spoke model and only pushing aggregated or required data through.

Why ADO is Required for V3 Apps

All V3 SPM apps are built on Polaris and designed for ADO from the ground up. Starting May 2026, when T&Q V3 ships, partners will need to:

  1. Set up an ADO instance for the customer
  2. Create source connections (Salesforce, Snowflake, flat files, etc.)
  3. Define transformation views for each key data object (accounts, employees, opportunities, territories)
  4. Create links from ADO views to the generated spoke model's import actions
  5. Set up ADO Workflows to schedule regular data pushes

The Anaplan Apps team is working toward auto-generating ADO links as part of app generation. Target: ADO links for all key spoke imports are generated automatically; partners only need to map the links to the customer's actual source views.

ADO Training
ADO training is available on Anaplan Academy and is worth the ~half-day investment before your first V3 project. Both Chan and John have completed it and recommend it. There's a noticeable learning curve, but the architecture is clean once it clicks.

Territory Data Structure Change (Important)

This is one of the most impactful changes between V2 and V3 for data preparation:

V2 vs V3 Territory File Format
V2: Provide one flat file with all hierarchy tiers as columns (T1, T2, T3, T4, T5 in one file). The Data Hub parses the hierarchy and creates the list structure.

V3: Provide a separate source view per tier in ADO. Each view only needs the member name and its parent. If you have a 5-tier hierarchy, you provide 5 ADO views.

Update your customer data prep conversations and the Data Prep Workbook guidance accordingly.

For each tier view in V3, you need at minimum:

  • Member name / ID
  • Parent member name / ID
  • Any additional reporting attributes for that tier

ADO Setup Tips

  • Connectors available today: Flat file, Salesforce, generic REST/JDBC. Snowflake connector is in progress.
  • Bidirectional feeds coming: Snowflake and Databricks connectors will support writing data back from Anaplan to the source — currently all ADO flows are one-directional (inbound to Anaplan).
  • Batch size limits: Be aware of record volume limits per push. Larger datasets may require batching strategies.
  • Performance tip: ADO can be slower than expected when inserting new list members (same as Polaris generally). Plan data load windows accordingly.
  • Workflow entitlement: Included with app subscription — customers do not need to purchase it separately. Confirm this is provisioned in their tenant before project kickoff.

Existing COE / Data Hub Customers

If a customer already has an established COE with a functioning Data Hub, they are not required to migrate to ADO immediately. The existing Data Hub can continue to feed the V2 spoke. Options to discuss with the customer:

  • Keep Data Hub (V2): Continue using the existing integration pattern. Straightforward, no additional setup. Limited to V2 app.
  • Migrate to ADO (V3): Required if they want V3 apps. Plan for a data integration migration project as part of the upgrade.
  • Run in parallel temporarily: Use Data Hub for current app while planning the ADO transition.
Don't Force ADO
Don't push ADO on a customer who has a well-functioning Data Hub setup, especially mid-project. The value proposition is clear for new implementations and V3 upgrades — but for an existing V2 customer, the migration cost may not be worth it until they're ready to upgrade to V3.
