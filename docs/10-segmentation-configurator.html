Account Segmentation & Scoring Configurator

Key configuration decisions for the Segmentation app

📖 Tell Segmentation Configuration You'll learn the key dimensionality decisions in the Segmentation configurator and how they cascade through the generated app.
👁 Show Facilitator Demo Instructor opens the Segmentation configurator and walks through each key setting, explaining the downstream impact of each choice.
✅ Do Segmentation Lab Complete the Segmentation section of the CloudWorks configurator exercise on the next lab page.

The Segmentation configurator shares the same app as Capacity. Navigate to the Segmentation section within the configurator. Key principle: you're defining at what level of detail you want to manage various assumptions. The dimensionality choices you make here cascade through the entire app - changing them post-generation is very painful.

Overview of Segmentation Configuration Goals

  • Define the account universe and what data objects drive segmentation
  • Configure scoring dimensions and wallet size calculation methods
  • Set up rule-based segmentation parameters
  • Define hierarchy planning setup (GTM modeling)
Complexity Reference: T&Q configurator = 1.0 complexity. Segmentation configurator = ~0.4 complexity. Fewer pages, fewer decisions, less dimensionality - but still requires care.

Page: Data Setup

This is a mirror of the T&Q data setup - same data objects (accounts, employees, opportunities). If you're configuring both Segmentation and Capacity, you only need to set this up once.

  • Additional segmentation-specific attributes: firmographic data - employee count bands, revenue bands, industry
  • Key: Enable list-formatted attributes (Enable for Calculate) for anything you want to use as a segmentation dimension
  • Banded attributes are required for the rule engine - numeric values don't work directly. For example: "employee band: 0-250, 250-1,000, 1,000+" rather than a raw number field

Page: GTM Performance Analysis

This page defines how historical and ongoing Go-to-Market performance data is structured and reported within the Segmentation app. Decisions made here determine the analytical backbone of the app's reporting capabilities.

  • Capture reporting dimensions - identify which dimensions (e.g., account segment, region, product line) GTM performance will be analyzed across. These drive the filter and aggregation options available to planners in the generated app.
  • Hierarchical reporting application - define how performance data rolls up through the hierarchy. Specify which levels are relevant for performance reporting and how parent levels aggregate child-level data.
  • Applicable record type for opportunity reporting - specify the Anaplan record type (e.g., Opportunity, Opportunity Line) used to source opportunity data into GTM performance analysis. This must align with the data objects configured in Data Hub Setup.
  • Data source mapping - document where GTM performance data originates (CRM system, flat file import, or connected Anaplan workspace) and how each source field maps to the Segmentation model's data structure.
📝 [SME CONFIRMATION NEEDED: GTM Performance Analysis - exact UX page name/path in Segmentation configurator]

Confirm the exact page name and location within the configurator for this section before the lab exercise. The data source (CRM, flat file, or connected workspace) for CloudWorks GTM data also needs to be confirmed before this section can be completed in a live session.

Page: Customer Profile Setup

The Customer Profile is a foundational input to Account Segmentation. It captures key account characteristics that inform which segment each account is assigned to. Without a correctly configured Customer Profile, the segment outputs will be based on incomplete or incorrect data.

  • Select drivers for customer profile - choose which account attributes (qualitative and quantitative, as classified in Sample Data) will drive the customer profile definition. The customer profile is used to inform the account segment - this is the direct input to the scoring and rule engine.
  • Select reporting dimension for customer profile - at what dimension level is the customer profile analyzed and reported? This determines how profile-level analysis is surfaced in the generated app (e.g., by region, by industry vertical, by account segment).
📝 [SME CONFIRMATION NEEDED: Customer Profile Setup - exact UX page name/path in Segmentation configurator]

Confirm the exact page name and navigation path before the lab exercise. The attribute list available for customer profile drivers will depend on what was enabled in Data Hub Setup.

Page: UCP / Account Hierarchy Configuration

🚨 Critical Terminology: UCP = Unacceptable Customer Profile

UCP stands for Unacceptable Customer Profile. Its purpose is to EXCLUDE accounts from the segmentation consideration - not to define which accounts are included. Accounts that meet UCP criteria are removed from the segmentation universe entirely and will not receive a segment assignment. Do not confuse UCP with an eligibility or inclusion filter.

  • Select account attributes relevant to UCP rules - identify which account attributes trigger exclusion. When an account matches the defined UCP attribute conditions, it is removed from segmentation consideration. Common examples: accounts below a minimum employee threshold, accounts in markets outside the GTM scope, or accounts flagged as inactive in the CRM.
  • Select account hierarchy alignment level - determine where Territory Hierarchy and Industry hierarchy alignment occurs. Specifically, call out cases where a child account differs from its parent account in hierarchy alignment - these accounts may need to be treated independently rather than inheriting the parent's segment.
  • Capture applicable account hierarchy level # - document the specific numeric hierarchy level used for alignment (e.g., Level 3 = Region, Level 5 = Territory). This numeric level selection directly affects how account segments flow into T&Q territory carving and Capacity headcount planning downstream.
📝 [SME CONFIRMATION NEEDED: UCP / Account Hierarchy Configuration - exact UX page name/path in Segmentation configurator; valid account hierarchy level # range]

Confirm the exact page name, navigation path, and the valid range of account hierarchy levels available for selection before the lab exercise.

Page: Account Hierarchy Setup

Define account hierarchy levels - typically from Global Ultimate Parent down through the subsidiary structure.

Critical decision: at what level do you segment?

  • Parent-level segmentation: All subsidiaries inherit the parent's segment. Simpler, fewer unique segment assignments.
  • Lowest-level segmentation: Each entity gets its own segment independently. More granular, but significantly more complex to manage.

This toggle is set in the configurator and is very painful to change post-generation. Be deliberate with the customer before committing.

Page: Rule-Based Segmentation Setup

Define which attributes are available as rule dimensions and how the rule-based scoring is structured.

  • Attributes must be list-formatted (configured in Data Setup) to be available as rule dimensions
  • Select account attributes for Rule-Based Segmentation from the list populated during Data Hub Setup - only enabled, list-formatted attributes will appear
🚨 Important: Segment Names and Count Are Not Configured Here

The number of segments and their names (e.g., Strategic, Named, Commercial, SMB) are not configured in the Configurator. They are defined by admins after application generation, via placeholder slots in the generated admin UX. No configuration action is needed here for segment naming or count.

Fewer segments is better. Customers often want 8-10 segments; push back toward 3-5. More segments = more complexity in every downstream app - more territory types, more coverage ratios, more headcount planning rows. Every extra segment you add compounds through T&Q and Capacity.

Page: Score-Based Segmentation Setup

Configure the scoring model that assigns every account a numeric score, which then maps to a segment.

  • Define up to 10 scoring metrics and their weights — weights must sum to 100%
  • Score range: 0–10 by default (can be changed, but all scoring must scale accordingly)
  • Define score-to-segment mapping thresholds (e.g., 7–10 = Strategic, 4–7 = Mid-Market, 0–4 = SMB)
  • Every account gets a score → every account gets a segment (unlike rule-based, which may leave gaps if an account doesn’t match any rule)
  • Rule-Based ALWAYS overrides Score-Based when both are available for the same account. Score-based acts as the default catch-all and only applies where no rule exists for that account.

Page: Wallet Size Configuration

Wallet size estimates the total addressable spend for each account - your potential revenue from that account. Three methods are available and can be used in combination, selectable per industry:

  1. OpEx method: Based on company operating expenditure as a percentage of their spend in your category
  2. Employee method: Based on spend per employee × employee count
  3. Revenue method: Based on company revenue × addressable market percentage
  • Geo index: Regional scaling factor - e.g., Americas gets a slight uplift, APAC a slight reduction
  • Remaining wallet = total wallet - current ARR = your potential / whitespace
  • This feeds directly into T&Q as the default allocation method for target cascading
Wallet size is optional. If the customer won't use it for target allocation or segmentation rules, skip the setup entirely. Don't over-engineer it. If they don't have reliable firmographic data and won't buy third-party data, the wallet size model will be directional at best.

Page: GTM Hierarchy Planning Setup

Define how many levels of GTM hierarchy planning you want to model. This is the sandbox for modeling next year's territory structure before committing anything to T&Q.

  • Completely separate from the live hierarchy - changes here don't affect T&Q until explicitly exported
  • The number of levels you configure here determines how the planning hierarchy section of the Segmentation app is built
  • V2 redesign (imminent): significantly more intuitive, with better hierarchy balance reporting

Timing Guidance

Configurator Estimated Time Relative Complexity
T&Q ~1 week 1.0×
Segmentation ~2-3 hours ~0.4×
Capacity ~2 days ~0.5×

Segmentation is much simpler than T&Q - fewer pages, less dimensionality. A well-prepared customer with clean firmographic data can move quickly. The biggest time sink is usually the wallet size configuration, which you can skip if it's not needed.
